# Plan: Store raw messages, generate HTML on-the-fly

## Overview
Currently the bot generates a complete Discord-themed HTML page and stores it in `transcripts.html_content`. Instead, store serialized message data and generate the HTML at view time on the dashboard.

## Changes needed

### Phase 1 ‚Äî Bot: Store serialized messages alongside HTML

#### 1. New table: `transcript_messages`
Add to `Management/gsrp-management/utils/database.js`:
```sql
CREATE TABLE IF NOT EXISTS transcript_messages (
    id INT AUTO_INCREMENT PRIMARY KEY,
    transcript_id VARCHAR(36) NOT NULL,
    message_data JSON NOT NULL,
    author_id VARCHAR(255) NOT NULL,
    created_timestamp BIGINT NOT NULL,
    sort_order INT NOT NULL,
    INDEX idx_transcript (transcript_id)
)
```
The `message_data` JSON column stores the full resolved message (see format below).

#### 2. Update `createTranscript` in `ticketUtils.js`
After the HTML is generated, serialize each Discord.js `Message` object into JSON and batch-insert into `transcript_messages`.

Serialized message format:
```json
{
  "id": "original-discord-message-id",
  "author": {
    "username": "User",
    "avatarUrl": "https://cdn.discordapp.com/...",
    "bot": false,
    "roleColor": "#ff0000",
    "roleName": "Admin",
    "roleIconUrl": null
  },
  "content": "Hello <@123456>",
  "timestamp": 1712345678000,
  "type": 0,
  "editedTimestamp": null,
  "replyTo": null,
  "attachments": [{"url": "...", "height": 100, "width": 200, "name": "file.png"}],
  "embeds": [
    {
      "title": "Embed Title",
      "description": "Embed description",
      "color": 16711680,
      "author": {"name": "Auth", "iconUrl": "..."},
      "fields": [{"name": "F1", "value": "V1", "inline": true}],
      "thumbnail": {"url": "..."},
      "image": {"url": "..."},
      "footer": {"text": "Footer", "iconUrl": "..."},
      "timestamp": "2024-01-01T00:00:00Z"
    }
  ],
  "components": [/* serialized ActionRow/Component structure */],
  "stickers": [{"url": "...", "name": "sticker"}],
  "mentions": {
    "users": {"123": {"name": "MentionedUser"}},
    "roles": {"456": {"name": "RoleName", "color": "#ff0000"}},
    "channels": {"789": {"name": "channel-name"}}
  }
}
```

Key serialization details:
- `msg.author` ‚Üí resolve `username`, `displayAvatarURL({ extension: 'png', size: 64, forceStatic: false })`, `bot`, `flags`
- `msg.member` ‚Üí resolve `displayName`, `roles.hoist.name`, `roles.hoist.iconURL()`, `roles.highest.hexColor`
- `msg.content` ‚Üí store RAW content (don't pre-format; the dashboard will handle markdown‚ÜíHTML)
- `msg.mentions` ‚Üí resolve user/role/channel mentions to lookups (name, color)
- `msg.embeds` ‚Üí serialize embed objects with all fields resolved
- `msg.components` ‚Üí serialize the component structure (ActionRow, Button, SelectMenu, etc.)
- `msg.attachments` ‚Üí serialize url, height, width, name
- `msg.stickers` ‚Üí serialize url, name

#### 3. DB function: `saveTranscriptMessages`
New function in `transcriptDB.js`:
```js
async function saveTranscriptMessages(transcriptId, messages) {
  const values = messages.map((msg, i) => [
    transcriptId,
    JSON.stringify(msg.data),
    msg.authorId,
    msg.timestamp,
    i
  ]);
  // batch insert with pool.query
}
```

### Phase 2 ‚Äî Dashboard: Create `lib/transcript-renderer.js`

A Node.js module that takes serialized messages and produces the same HTML format the bot currently generates. This lives in the dashboard codebase since Vercel will execute it at request time.

#### Exported function:
```js
export function generateTranscriptHTML({ messages, channelName, closedAt, reason, openerTag, openReason, extraFields, staffRequestReason }) {
  // returns the full HTML string
}
```

#### Implementation mirrors `ticketUtils.js`:
1. **`buildProfiles(messages)`** ‚Äî iterate messages, collect unique authors, return profiles map
2. **`formatMessageContent(content, mentions, profiles)`** ‚Äî port the bot's `formatMessageContent` to use the serialized mentions/profiles instead of Discord.js objects:
   - Escape HTML
   - Code blocks ‚Üí `<discord-code-block>`
   - Inline code ‚Üí `<discord-code>`
   - User mentions ‚Üí look up in `mentions.users` / `profiles` ‚Üí `<discord-mention type="user">`
   - Role mentions ‚Üí look up in `mentions.roles` ‚Üí `<discord-mention type="role">`
   - Channel mentions ‚Üí look up in `mentions.channels` ‚Üí `<discord-mention type="channel">`
   - Timestamps ‚Üí `<discord-time>`
   - Custom emoji ‚Üí `<discord-custom-emoji>`
   - Bold/italic/underline/strikethrough ‚Üí `<discord-bold>` etc.
   - Newlines ‚Üí `<br>`
3. **`renderEmbeds(embeds)`** ‚Äî convert serialized embeds to `<discord-embed>` tags
4. **`renderAttachments(attachments)`** ‚Äî convert to `<discord-attachment>` tags
5. **`renderComponents(components)`** ‚Äî convert to `<discord-action-row>`, `<discord-button>`, etc.
6. **`renderMessages(messages, profiles)`** ‚Äî wrap each message in `<discord-message>` with profile, timestamp, content, embeds, attachments, components, reply info
7. **`buildHTML(template)`** ‚Äî assemble the full HTML document with:
   - `<!DOCTYPE html>` wrapper
   - `<script id="discord-profiles-data">` with profiles JSON
   - `<script>` loading `@derockdev/discord-components-core` from CDN
   - `<discord-messages>` container
   - Meta info section (channel name, date, reason, etc.)

#### Use existing dashboard code:
- `escHtml` already exists in `pages/transcript/[id].js` ‚Äî extract to shared lib or keep in renderer
- `renderDiscordMarkdown` is NOT used here ‚Äî the renderer outputs `<discord-*>` web component tags (same as bot), the dashboard's span-based markdown is only for non-message text

### Phase 3 ‚Äî Dashboard: Update viewer page

#### 3a. New API route: `pages/api/transcripts/[id]/render.js`
Accept `POST` or `GET` with transcript ID, return the generated HTML.
Or simpler: do the generation inside `getServerSideProps` directly (no extra API call).

#### 3b. Update `getServerSideProps` in `pages/transcript/[id].js`
```js
// Try to render from messages first
try {
  const [msgRows] = await pool.query(
    'SELECT message_data, author_id, created_timestamp FROM transcript_messages WHERE transcript_id = ? ORDER BY sort_order ASC',
    [id]
  );
  if (msgRows.length > 0) {
    const messages = msgRows.map(r => JSON.parse(r.message_data));
    const htmlContent = generateTranscriptHTML({
      messages,
      channelName: t.channel_name,
      closedAt: t.closed_at,
      reason: t.close_reason,
      // ... other metadata
    });
    return { props: { htmlContent, id, meta, canManage } };
  }
} catch (e) {
  if (e.code !== 'ER_NO_SUCH_TABLE') throw e;
}
// Fallback to stored HTML
// ... existing code that uses t.html_content
```

## Message format compatibility
Once the bot saves serialized messages AND the dashboard can render them, the stored `html_content` column becomes redundant for new transcripts. Existing transcripts that only have `html_content` continue to work via the fallback path.

## Migration (optional)
Write a Node.js script that:
1. Queries transcripts WHERE `html_content IS NOT NULL`
2. Uses cheerio to parse the stored HTML (extract messages from `<discord-message>` elements)
3. Reconstructs message JSON from the HTML
4. Saves to `transcript_messages`
This is lossy (no raw mentions/embeds/components) but gives the fallback-free path.

## Key risks
1. **Vercel 10s timeout**: Generate HTML synchronously in `getServerSideProps`. For transcripts with 1000+ messages, test if it stays under limit. Mitigation: keep `html_content` fallback.
2. **Discord markdown differences**: Ensure `formatMessageContent` in the dashboard exactly matches the bot's output. Test with various message types.
3. **Mention resolution**: The dashboard doesn't have access to Discord.js guild caches. All mention data must be pre-resolved by the bot and stored in the serialized data.
4. **Web component version matching**: Both bot-generated HTML and dashboard-generated HTML use the same CDN version (`@derockdev/discord-components-core@^3.6.1`).
